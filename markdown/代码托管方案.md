# 小白可执行的 Xboard 官方更新 + 自定义插件/主题维护方案

## Summary

- 推荐做一个你自己的私有仓库 `xboard-custom`，但它**不是**作者仓库的长期 fork。
- `xboard-custom` 只维护你自己的内容：3 个插件、1 个自定义主题、打包脚本、更新说明。
- 生产站继续用官方 1Panel / Docker Compose 方式跑官方镜像；官方更新走作者流程，自定义更新走你自己的插件/主题版本包。
- 只有将来你必须改官方核心目录 `app/`、`config/`、`routes/`、内置 `theme/` 时，才需要把整个 Xboard fork 成长期自维护版本。

## Repo And Directory Layout

- 官方运行目录：
  - 例子：`/opt/1panel/apps/openresty/openresty/www/sites/xboard/index`
  - 这里放官方 `compose.yaml`、`.env`、运行中的 `plugins/`、`storage/theme/`、`storage/logs/`
  - Compose 结构按 [Xboard-compose/compose.yaml](D:\codex\Xboard-master\Xboard-compose\compose.yaml) 或 [1panel.md](D:\codex\Xboard-master\docs\en\installation\1panel.md) 保持 `plugins` 和 `storage/theme` 为持久挂载目录

- 你的私有仓库 `xboard-custom`：
  - `plugins-src/StripeCheckout/`
  - `plugins-src/BEpusdt/`
  - `plugins-src/WalletCenter/`
  - `theme-src/XboardCustom/`
  - `release/plugins/*.zip`
  - `release/themes/*.zip`
  - `docs/DEPLOY.md`
  - `docs/UPDATE.md`

- 版本规则：
  - 每个插件自己的 `config.json` 维护版本号，如 `1.0.0`
  - 自定义主题自己的 `config.json` 也维护版本号
  - 后续更新时必须先涨版本，再重新打包 zip；Xboard 的插件/主题上传更新机制就是靠版本号判断新旧

## First Deployment

1. 在 1Panel 按官方文档部署 Xboard，先不要放任何自定义代码。
2. 站点目录准备好以下持久目录：
   - `.env`
   - `.docker/.data`
   - `storage/logs`
   - `storage/theme`
   - `plugins`
3. 启动官方服务并完成初始化安装：
   ```bash
   docker compose up -d
   docker compose run --rm web php artisan xboard:install
   ```
4. 登录后台，只先做基础配置：
   - 站点域名
   - 数据库
   - Redis
   - 邮件
   - 基础管理员账号
5. 开发完成后，不要直接去改官方运行目录源码；把自定义内容在 `xboard-custom` 仓库里打成发布包：
   - `StripeCheckout-1.0.0.zip`
   - `BEpusdt-1.0.0.zip`
   - `WalletCenter-1.0.0.zip`
   - `XboardCustom-1.0.0.zip`
6. 生产站安装自定义内容时，优先使用后台上传：
   - 插件：后台插件管理上传 zip，然后安装、启用、配置
   - 主题：后台主题管理上传 zip，然后切换到 `XboardCustom`
7. 支付相关配置只在后台填：
   - Stripe 的 Secret Key、Webhook Secret
   - BEpusdt 的 Base URL、API Token、法币/网络限制
8. 验证上线前必须做一轮完整测试：
   - Stripe 下单、支付成功、回调成功
   - BEpusdt 下单、支付成功、回调成功
   - 签到成功且同日不能重复
   - 余额充值成功到账
   - 自动续费开关、到期前 24 小时续费逻辑
   - 主题切换后页面和多语言正常

## Update Workflow

### A. 更新作者官方 Xboard

1. 更新前先备份：
   - 数据库
   - `.env`
   - 当前已用的插件配置截图
   - 当前主题配置截图
   - 你自己的 `release/` 发布包
2. 在站点目录执行官方更新：
   ```bash
   docker compose pull
   docker compose run --rm web php artisan xboard:update
   docker compose up -d
   ```
3. 官方更新完成后，先不要急着改配置，先做基础巡检：
   - 能否登录前后台
   - 支付方式列表是否还在
   - 插件是否仍显示启用
   - 当前主题是否仍是 `XboardCustom`
4. 因为 `plugins/` 和 `storage/theme/` 是挂载目录，正常情况下你的自定义插件和主题不会被官方更新覆盖。
5. 如果官方更新后出现兼容问题，不要先改线上官方代码；应在 `xboard-custom` 仓库修兼容、涨版本、重新打包，再通过后台上传更新你的插件或主题。

### B. 更新你自己的插件和主题

1. 在 `xboard-custom` 仓库修改源码。
2. 修改对应 `config.json` 版本号。
3. 重新打包 zip 放进 `release/`。
4. 生产站更新方式：
   - 插件：后台再次上传新 zip，触发插件 update 逻辑和插件 migration
   - 主题：后台再次上传新 zip，触发主题 update 逻辑并保留已有主题配置
5. 更新后验证本次改动相关功能，不要只看“上传成功”。

## Operational Interfaces

- 你的长期维护接口不是“直接改线上文件”，而是这 3 个固定入口：
  - 官方更新入口：`docker compose pull` + `php artisan xboard:update`
  - 自定义插件更新入口：后台上传新版插件 zip
  - 自定义主题更新入口：后台上传新版主题 zip
- 你的长期版本管理对象不是整套 Xboard，而是：
  - `release/plugins/*.zip`
  - `release/themes/*.zip`
  - 更新文档和版本记录
- 生产环境禁止使用 [update.sh](D:\codex\Xboard-master\update.sh) 这种会 `git reset --hard` 的脚本，因为它适合整仓 Git 覆盖更新，不适合你这种“官方系统 + 自定义挂载内容”的模式。

## Rollback And Safety

- 每次官方更新前保留一份数据库备份和当前自定义发布包。
- 每次自定义更新前保留上一版插件 zip 和主题 zip。
- 如果官方更新后出问题：
  - 先回滚数据库和站点镜像版本
  - 不要在线上临时改核心代码救火
  - 在 `xboard-custom` 仓库修复后重新发布
- 如果自定义插件更新后出问题：
  - 直接上传上一版插件 zip 回退
- 如果自定义主题更新后出问题：
  - 直接上传上一版主题 zip 或切回系统默认主题

## Assumptions

- 维护方式采用你已选的“混合模式”。
- 官方 Xboard 继续使用 1Panel + Docker Compose 部署。
- 你自己的仓库默认用私有仓库。
- 自定义内容全部限制在插件和自定义主题内，不改官方核心目录。
- 后续如果业务需求迫使修改官方核心，再单独升级为“完整 fork + upstream 合并”的维护模式。
