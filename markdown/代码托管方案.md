# Xboard 官方仓库 + 自定义仓库代码托管方案

## Summary

- 官方仓库 `https://github.com/cedar2025/Xboard` 只负责提供和更新 Xboard 底座。
- 自定义仓库 `https://github.com/ksahdsambn/xboard-custom` 是唯一的自定义源码真源，只维护插件、主题和文档。
- 生产环境不直接长期修改官方仓库，也不把官方仓库 fork 成你的日常开发主仓库。
- 生产部署采用 overlay 模式：把 `xboard-custom` 中的自定义目录同步到官方运行目录。

## 当前仓库结构

- 官方运行目录示例：
  - `/opt/1panel/apps/openresty/openresty/www/sites/xboard/index`
  - 这里保存官方 `compose.yaml`、`.env`、`plugins/`、`storage/theme/`、`storage/logs/`
- 自定义仓库 `xboard-custom`：
  - `plugins/StripePayment/`
  - `plugins/BepusdtPayment/`
  - `plugins/WalletCenter/`
  - `theme/XboardCustom/`
  - `scripts/deploy-overlay.sh`
  - `markdown/DEPLOY.md`
  - `markdown/progress.md`
  - `markdown/architecture.md`
  - `markdown/bug_fix.md`

## 为什么采用 overlay 模式

- 最稳定：不直接侵入官方核心目录 `app/`、`config/`、`routes/`、`resources/`
- 最便捷：官方更新和自定义更新分开处理
- 最符合开发规范：官方底座与自定义源码职责清晰
- 最不怕覆盖：官方更新后，只需重新同步一次自定义层

## 唯一推荐的托管边界

- 官方仓库继续按作者文档在 1Panel 中安装和升级。
- 你的 GitHub 仓库只放以下自定义资产：
  - `plugins/StripePayment/`
  - `plugins/BepusdtPayment/`
  - `plugins/WalletCenter/`
  - `theme/XboardCustom/`
  - `markdown/*.md`
  - `scripts/deploy-overlay.sh`
- 不把以下官方核心目录纳入你的自定义仓库：
  - `app/`
  - `bootstrap/`
  - `config/`
  - `database/`
  - `public/`
  - `resources/`
  - `routes/`

## 生产环境目录约定

- 官方运行目录：`/opt/1panel/apps/openresty/openresty/www/sites/xboard/index`
- 自定义仓库目录：`/opt/xboard-custom`

在官方 1Panel compose 结构下，关键挂载点是：

- `plugins/` -> 容器 `/www/plugins`
- `storage/theme/` -> 容器 `/www/storage/theme`

因此发布时固定同步到：

- `xboard-custom/plugins/*` -> 官方运行目录 `plugins/*`
- `xboard-custom/theme/XboardCustom` -> 官方运行目录 `storage/theme/XboardCustom`

不要把自定义主题作为长期维护版本直接放进官方运行目录根下的 `theme/XboardCustom`。如果服务器里残留该目录，应当清理，避免主题优先级冲突。

## 发布与更新入口

长期维护只保留两个固定入口：

1. 官方底座更新入口
   - 在官方运行目录执行作者仓库更新
   - 再执行 `php artisan xboard:update`
2. 自定义层更新入口
   - 在 `xboard-custom` 仓库 `git pull`
   - 再执行 `scripts/deploy-overlay.sh`

推荐脚本：

- 官方更新后重新叠加自定义层：
  - `cd /opt/1panel/apps/openresty/openresty/www/sites/xboard/index`
  - `git pull --ff-only origin compose`
  - `docker compose pull`
  - `docker compose run --rm -T web php artisan xboard:update`
  - `docker compose up -d`
  - `cd /opt/xboard-custom`
  - `git pull`
  - `OFFICIAL_ROOT=/opt/1panel/apps/openresty/openresty/www/sites/xboard/index bash scripts/deploy-overlay.sh`
- 仅更新自定义层：
  - `cd /opt/xboard-custom`
  - `git pull`
  - `OFFICIAL_ROOT=/opt/1panel/apps/openresty/openresty/www/sites/xboard/index bash scripts/deploy-overlay.sh`

## 后台与版本管理原则

- 插件是否安装、启用、升级，仍通过后台 `插件管理` 操作。
- 当前主题是否为 `XboardCustom`，仍通过后台 `主题管理` 检查和切换。
- 如果某个插件 `config.json` 的版本号提高了，同步代码后还需要在后台点一次对应插件的 `升级`。
- 主题发布主流程不是后台上传 zip，而是仓库同步覆盖到 `storage/theme/XboardCustom`。

## 关于 zip 包的定位

- zip 包不再是主发布方式。
- zip 只作为可选的应急分发方式或线下备份方式。
- 如果未来确实需要交付离线安装包，可以单独从当前仓库打包：
  - `plugins/StripePayment`
  - `plugins/BepusdtPayment`
  - `plugins/WalletCenter`
  - `theme/XboardCustom`
- 但只要服务器允许 `git pull`，生产环境长期维护仍应优先使用 overlay 同步。

## 回滚与安全策略

- 每次官方更新前备份数据库和 `.env`
- 每次大版本更新前记录后台插件配置和主题状态
- 若官方更新后异常：
  - 先确认官方仓库是否更新成功
  - 再重新执行一次 overlay 同步
  - 仍有问题时，在 `xboard-custom` 仓库修复并重新同步
- 若只因自定义层更新导致异常：
  - 在 `xboard-custom` 回退到上一提交
  - 重新执行 `scripts/deploy-overlay.sh`

## 何时才需要 fork 官方仓库

只有在下面情况出现时，才需要考虑把官方仓库升级为长期 fork：

- 必须修改 `app/`
- 必须修改 `config/`
- 必须修改 `routes/`
- 必须修改官方内置 `theme/Xboard/`
- 必须修改 compose 之外的官方核心运行逻辑

在当前项目范围内，不需要这么做。

## 结论

当前项目最合适的代码托管方式只有一套：

- 官方仓库负责底座
- `xboard-custom` 负责自定义源码
- 服务器通过 `deploy-overlay.sh` 把自定义层同步进官方运行目录
- 1Panel 只负责可视化管理站点、任务、数据库和容器

这套结构和 [DEPLOY.md](D:\codex\xboard-custom\markdown\DEPLOY.md) 保持一致，也和当前仓库实际目录保持一致。
