# Xboard 可升级扩展 AI 开发任务清单

## 1. 使用说明

本清单面向 AI 开发者，用于在无人值守条件下执行 Xboard 扩展开发任务。

执行时必须遵守以下规则：

- 必须以 `requirements.md` 作为需求边界依据。
- 必须以 `implementation-plan.md` 作为实施路径依据。
- 必须按顺序执行，不得跳步。
- 每一阶段的“强制测试”全部通过后，才能进入下一阶段。
- 任一阶段出现失败项，必须先修复并重新验证，不得带着失败项进入后续阶段。
- 必须优先采用插件和自定义主题扩展，不得默认修改官方核心。
- 必须记录所有发现的边界冲突、兼容性风险和回归风险。
- 必须确保所有资金相关流程具备幂等性。

## 2. 全局质量门槛

在开始逐项开发前，先确认以下全局目标：

- [ ] 仅支持 `Xboard` 前台主题。
- [ ] 不修改管理员后台语言。
- [ ] Stripe 只做一次性支付。
- [ ] BEpusdt 只做一次性支付。
- [ ] 每日签到面向所有已登录用户。
- [ ] 余额充值采用自定义金额。
- [ ] 自动续费只使用余额。
- [ ] 自动续费按当前订阅周期执行。
- [ ] 自动续费在到期前 24 小时尝试。
- [ ] 新增 13 种语言覆盖整个 `Xboard` 前台主要页面与主要交互文案。
- [ ] 所有新增功能都必须保留后续跟进官方更新的能力。

## 3. 阶段一：项目基线分析

- [ ] 确认主题切换机制、自定义主题加载方式和主题配置继承方式。
- [ ] 确认用户余额、订单余额抵扣、续费判断和用户状态相关逻辑。
- [ ] 列出本次开发允许触达的目录和应尽量避免触达的目录。
- [ ] 列出所有会被本次需求影响的业务路径。

### 强制测试

- [ ] 验证项目中存在独立支付插件接入点。
- [ ] 验证项目中存在自定义主题切换能力。
- [ ] 验证项目中存在插件计划任务接入能力。
- [ ] 验证普通订单流程与用户余额流程可以被清晰区分。
- [ ] 验证无需默认依赖大规模核心重构即可覆盖本次需求。

### 放行条件

- [ ] 已确认最小侵入式扩展路径。
- [ ] 已确认本次开发的关键依赖点和风险点。

- [ ] Stripe 插件完整可配置。
